# ADR-0006: 高リスク投稿には確認フラグを持たせる

- 日付: 2026-05-27
- ステータス: 承認済み

---

## コンテキスト

kyusoku はリスクを検知して管理画面に届けるが、「届けるだけ」では不完全だ。アプリがSOSを検知しても、担当営業が確認したかどうかが分からなければ、声は届いたことにならない。

特に、担当営業が多忙で管理画面を開いていない場合、高リスクの投稿が未確認のまま放置されるリスクがある。確認した記録が残る仕組みがなければ、「見た・見ていない」の曖昧さが生まれ、本来あるべき対応が起きなかったことも追跡できなくなる。

## 決定

警告（レベル2）・SOS（レベル3）の投稿には確認フラグを持たせる。

フラグの状態は以下の4段階とする。

| 状態 | 説明 |
|------|------|
| 未確認 | 誰もまだ確認していない |
| 確認済み | 担当営業が確認した |
| 対応中 | 担当営業が対応を開始した |
| 対応済み | 対応が完了した |

管理画面には未確認のまま経過した時間を表示する。

## 根拠

- SOSを届ける仕組みと、SOSが受け取られたことを確認する仕組みはワンセットである
- 「見た」という記録が残るだけで、担当営業の意識は変わる
- 完璧な運用は期待しない。それでも「記録が残ること」自体に意味がある
- アプリにできることは「届けて、確認を促すこと」まで。最後の一歩は人が動く

## 影響

- `risk_flags` テーブルに状態カラムと更新日時・更新者が必要になる
- 管理画面に確認操作のUIが必要になる
- 未確認の経過時間を表示するためのクエリをバッチで持つか、フロントで計算するかは実装時に決める
- フラグが動かない状態が長く続いても、アプリは警告を出す以上のことはしない。それはアプリの責任範囲の外にある
