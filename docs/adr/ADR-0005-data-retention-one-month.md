# ADR-0005: データ保持は1ヶ月とし、削除前にサマリーに畳む

- 日付: 2026-05-27
- ステータス: 承認済み

---

## コンテキスト

kyusoku は派遣社員の日々の気持ちを記録する。記録には個人の生の言葉が含まれるため、保持期間が長いほどプライバシーのリスクが高まる。特に「退職後も個人の言葉が残り続ける」ことは、倫理的に問題がある。

一方で、傾向分析や離職率との相関を見るためには、一定期間の集計データは必要だ。生の言葉と集計値を分けて管理することで、この二つの要求を両立できる。

## 決定

`daily_reports` と `access_logs` は1ヶ月で削除する。削除前に `monthly_summaries` テーブルに集計して畳み込む。

月次バッチの処理順序を以下に固定する。

```
月が変わったら
  1. 前月のrawデータを集計してmonthly_summariesに書き込む
  2. 前月のdaily_reports・access_logsを削除する
```

`monthly_summaries` は月単位で洗い替えとし、個人の生の言葉は含めない。スコアと集計値のみを保持する。

サマリーに含む情報は以下の通り。

| 項目 | 説明 |
|------|------|
| 気分スタンプ月間分布 | 元気・普通・悲しい の件数 |
| 提出率 | 提出日数 / 稼働日数 |
| リスク判定件数 | レベル別の件数 |
| 沈黙最大継続日数 | その月で最も長く途切れた日数 |
| 記述あり提出件数 | テキストを添えて送信した日数 |
| アクセス日数 | アプリを開いた日数 |

## 根拠

- 派遣社員が退職した後も個人の言葉が残り続けることは、このアプリの倫理的な姿勢と一致しない
- プライバシーが時間とともに自然に薄れていく設計にすることで、「いつまでも見られている」という感覚を残さない
- サマリーは個人の言葉を持たないため、長期保持しても倫理的に問題が少ない
- D1（SQLite）の容量に対しても、1ヶ月のrawに限定することで負荷を小さく保てる

## 影響

- 月次バッチWorkerの実装が必要になる
- `monthly_summaries` のスキーマ設計は、将来の集計ニーズを見越して決める必要がある
- 1ヶ月より古いデータへの細かいアクセスはできなくなる（設計上の意図的な制約）
- 管理画面は1ヶ月以内はrawから、それより古い期間はsummaryから表示する切り替えが必要
