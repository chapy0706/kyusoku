# ADR-0010: v1はプッシュ通知を実装しない

- 日付: 2026-05-27
- ステータス: 承認済み

---

## コンテキスト

高リスクの投稿が来たとき、担当営業にすぐ知らせる手段としてプッシュ通知やSlack連携が候補として挙がった。「SOSを届ける」というアプリの目的を考えると、通知の仕組みは自然な拡張方向だ。

しかし、通知の実装はアプリの複雑さを大きく増す。通知先の設定、Slack Webhookの管理、メール送信の実装、通知が届かなかった場合のフォールバック——これらを正しく作ろうとすると、コア機能が完成する前にスコープが膨らむリスクがある。

## 決定

v1ではプッシュ通知・Slack連携・メール通知を実装しない。担当営業が自分で管理画面を確認する運用とする。

「作れなかった」は本末転倒。まずコアを動かすことを優先する。

将来的なSlack連携は検討候補として残す。着手する際は新ADRを立てる。

## 根拠

- 通知の実装コストとv1の価値は釣り合わない
- 管理画面に確認フラグ（ADR-0006）と未確認経過時間が表示される設計があれば、能動的に確認する運用でも最低限機能する
- プッシュ通知は「やらされてる感」を育てやすい。信頼が薄れている現場では逆効果になるリスクもある
- シンプルに作り、まず動かして、知見を得てから育てる

## 影響

- v1の実装スコープが絞られ、コア機能の完成が早くなる
- 担当営業が管理画面を「自分で開く」運用になるため、習慣化の促しは別の手段（オンボーディングなど）で補う
- Slack連携を将来追加する際は、Incoming Webhookをインターフェース越しに扱い、ドメインがSlackを直接知らない設計にする
