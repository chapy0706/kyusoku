# ADR-0004: 沈黙の検知はルールベースで行い、AIを使わない

- 日付: 2026-05-27
- ステータス: 承認済み

---

## コンテキスト

kyusoku が拾いたいリスクのひとつは「書かなくなること」「来なくなること」だ。明確にネガティブな言葉を書く人より、静かにフェードアウトしていく人の方が多く、そして見つけにくい。

この「沈黙の検知」をAIで行うことは、原理的にできない。AIに渡せるのは「書かれた言葉」だけだ。書かれなかったものは、AIに渡すテキストが存在しない。沈黙はAIが読むものではなく、データの「無さ」を数える処理だ。

## 決定

沈黙の検知はルールベース（SQL + Cloudflare Workers Cron Triggers）で行う。AIは使わない。

検知するシグナルと計算方法を以下に定義する。

| シグナル | 計算方法 |
|----------|----------|
| スタンプ下降 | 直近7日の気分スタンプ平均の変化 |
| 悲しい連続 | 悲しいスタンプが何日連続したか |
| 提出途切れ | 最終提出日から現在までの日数差 |
| 記述消失 | 以前よく書いていたのに書かなくなった（has_note 比率の変化） |
| 未アクセス | アプリを開いていない日数 |

これらを `SilenceAnalyzer` として実装し、AIが出す `TextRiskClassifier` のスコアと合流して最終的なリスク評価を生成する。

## 根拠

- 沈黙は「書かれなかったこと」であり、テキスト処理の対象にできない
- ルールベースの方がAIより速く、安定していて、コストもかからない
- 50人規模の日次バッチであれば Cloudflare Workers の無料枠で十分に収まる
- AIに依存しないことで、沈黙検知はAIの精度に左右されない独立したシグナルになる

## 影響

- `daily_reports` に加えて `access_logs` テーブルが必要になる（アクセス日数の計測のため）
- `daily_reports` に `has_note` カラムが必要になる（記述消失の検知のため）
- `SilenceAnalyzer` と `TextRiskClassifier` は独立したコンポーネントとして実装し、`RiskAssessment` で合流させる
- 書いていない人でも沈黙スコアは出る。テキストスコアがゼロでも見逃されない構造になる
